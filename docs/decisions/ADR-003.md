**Date:** 2026-03  
**Status:** Accepted  
**Decides:** How money is represented in the database and application code

---

## Context

The platform handles monetary values in multiple places: product prices, GST amounts, delivery charges, platform commission, supplier payouts, and TCS deductions.

A decision was needed on how to store and compute these values consistently across the entire system.

---

## Decision

All monetary values are stored as **integers representing paise**.

₹1.00 = `100` in the database  
₹499.00 = `49900` in the database  
₹1,249.50 = `124950` in the database

This applies to every monetary column in every table. Column names are suffixed with `_paise` to make this explicit (e.g. `base_price_paise`, `gst_amount_paise`, `commission_paise`).

---

## Reasoning

### Floating point is unsafe for money

JavaScript's `number` type (IEEE 754 double-precision float) cannot represent many decimal values exactly:

```
0.1 + 0.2 === 0.30000000000000004  // true in JS
```

In financial calculations this causes rounding errors that compound. A GST calculation on ₹999 at 18% using floats may produce ₹179.82000000000002 — which when summed across thousands of orders produces real discrepancies in GST return filings.

### Integer arithmetic is exact

Paise are the smallest unit of Indian currency. All real monetary values in INR are expressible as an integer number of paise. Integer addition, subtraction, and comparison in both JavaScript and PostgreSQL are exact with no rounding.

### PostgreSQL `numeric` is an alternative but slower

PostgreSQL's `NUMERIC` / `DECIMAL` type is exact but:

- Slower than integer arithmetic for large aggregations
- Requires explicit precision/scale decisions
- More complex to handle in JavaScript (no native decimal type)

Integer paise avoids all of this.

---

## Rules for all engineers

**Always store paise. Always compute in paise. Convert to rupees only at the API response layer for display.**

```typescript
// CORRECT — store and compute in paise
const priceInPaise = 49900;           // ₹499.00
const gstPaise = Math.round(priceInPaise * 0.18);  // ₹89.82 → 8982 paise
const totalPaise = priceInPaise + gstPaise;        // 58882 paise

// CORRECT — convert to rupees only in the API response formatter
const display = {
  price: (priceInPaise / 100).toFixed(2),   // "499.00"
  gst: (gstPaise / 100).toFixed(2),         // "89.82"
  total: (totalPaise / 100).toFixed(2),     // "588.82"
};

// WRONG — never store or compute in rupees
const price = 499.00;  // floating point — do not do this
```

**Use `Math.round()` when computing fractional paise.** GST rates can produce fractional paise (18% of ₹499 = ₹89.82 = 8982 paise exactly — this one is fine). But rates like 5% of ₹333 = ₹16.65 = 1665 paise (fine). Always `Math.round()` the result to stay in integers.

---

## Consequences

- Every monetary column name must end in `_paise`
- API responses convert to rupees in a shared formatter function — never inline in business logic
- Any engineer used to storing money as `DECIMAL(10,2)` will find this unusual — this ADR is the explanation
