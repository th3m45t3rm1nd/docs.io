**Decides:** How product data changes in PostgreSQL are reflected in OpenSearch

---

## Context

The platform uses OpenSearch for product search. OpenSearch is a separate system from PostgreSQL. When a supplier updates a product price, changes stock availability, or adds a new listing, that change must be reflected in search results.

The question is: how do we keep OpenSearch consistent with PostgreSQL without losing updates?

---

## The problem with the naive approach

The obvious solution is to update both systems in the same API request:

```typescript
// NAIVE â€” do not do this
await db.query(`UPDATE supplier_products SET price_paise = $1 WHERE id = $2`, [newPrice, id]);
await opensearch.update({ index: 'products', id, body: { price_paise: newPrice } });
```

This has a critical flaw: if the OpenSearch call fails (network timeout, OpenSearch temporarily down, rate limit), the price is updated in PostgreSQL but NOT in search results. The discrepancy is silent â€” no error is shown to the user, no alert fires â€” and search results show the wrong price until the next manual re-index.

In a marketplace this is a trust and legal problem. A customer sees â‚ą499 in search results but the checkout price is â‚ą599.

---

## Decision

Use the **transactional outbox pattern**.

When a product or supplier_product row changes, a row is written to the `search_sync_outbox` table **in the same database transaction** as the product update. A separate background worker polls this outbox and sends updates to OpenSearch, then marks the row as processed.

```
API request
    â”‚
    â”śâ”€â”€ UPDATE supplier_products ... (in one transaction)
    â””â”€â”€ INSERT INTO search_sync_outbox ...

Background worker (every 5 seconds)
    â”śâ”€â”€ SELECT unprocessed rows FROM search_sync_outbox
    â”śâ”€â”€ Send each to OpenSearch
    â””â”€â”€ UPDATE search_sync_outbox SET processed_at = now()
```

---

## Why this works

The database transaction is atomic. Either both the product update AND the outbox insert succeed, or neither does. There is no state where the product changed but no sync record exists.

If OpenSearch is down, outbox rows accumulate. When it recovers, the worker drains the queue. OpenSearch may lag by seconds or minutes during an incident, but it will always eventually converge to the correct state.

PostgreSQL is always the source of truth. If OpenSearch and PostgreSQL ever disagree, PostgreSQL wins.

## Outbox table schema

```sql
CREATE TABLE search_sync_outbox (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  entity_type  text NOT NULL,  -- 'product', 'supplier_product'
  entity_id    uuid NOT NULL,
  operation    text NOT NULL,  -- 'upsert', 'delete'
  payload      jsonb,          -- pre-computed document to index
  created_at   timestamptz DEFAULT now(),
  processed_at timestamptz,    -- null = pending
  failed_at    timestamptz,    -- null = not failed
  retry_count  integer DEFAULT 0,
  error        text            -- last error message if failed
);

CREATE INDEX idx_outbox_pending
  ON search_sync_outbox (created_at)
  WHERE processed_at IS NULL AND failed_at IS NULL;
```

---

## Worker behaviour

The `search-sync` worker (in `src/workers/search-sync.ts`):

1. Polls for unprocessed rows every 5 seconds
2. Processes in batches of 50 (OpenSearch bulk API)
3. On success: sets `processed_at = now()`
4. On failure: increments `retry_count`, sets `failed_at`, stores error message
5. Retries failed rows up to 5 times with exponential backoff
6. Rows with `retry_count >= 5` are dead-lettered â€” an alert fires and ops must investigate manually

## What triggers an outbox entry

|Event|Entity type|Operation|
|---|---|---|
|Supplier creates a product listing|`supplier_product`|`upsert`|
|Supplier updates price or stock|`supplier_product`|`upsert`|
|Supplier deletes a listing|`supplier_product`|`delete`|
|Admin updates product name/attributes|`product`|`upsert`|
|Admin deactivates a product|`product`|`delete`|
|Supplier account is deactivated|`supplier_product`|`delete` (all)|

## Consequences

- OpenSearch is **eventually consistent** with PostgreSQL â€” there is a small lag (typically under 10 seconds in normal operation). This is acceptable for search.
- Checkout pricing always reads from PostgreSQL directly, never from OpenSearch â€” so the eventual consistency does not affect order correctness, only search result display.
- The `search-sync` worker must always be running. Monitor its queue depth â€” growing queue = worker is down or OpenSearch is unhealthy.
- A full re-index script exists for disaster recovery: `npm run reindex` truncates the outbox and inserts a row for every active product. Use only in emergencies.
