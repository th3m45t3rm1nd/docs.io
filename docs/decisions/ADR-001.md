**Status:** Accepted  
**Decides:** Language, framework, database, cache, search engine

## Context

We are building a construction materials and services marketplace for Delhi NCR. At the time of this decision:

- Engineering team is one backend/DevOps engineer
- We need to move fast without sacrificing correctness on financial logic
- Every major Indian third-party integration (Razorpay, MSG91, WhatsApp Business API, NIC e-way bill) must be supported
- Infrastructure must stay in India for data residency reasons
- The system must be maintainable by a future engineer who joins with standard Node.js or backend experience

---

## Decisions and reasoning

### Language: Node.js 20 with TypeScript

**Chosen over:** Go, Python, Java

Node.js was chosen primarily because:

- The entire Indian payments/SMS/logistics ecosystem has first-class Node.js SDKs. Razorpay, MSG91, Firebase all have official Node packages with active maintenance.
- TypeScript gives compile-time safety on the financial logic (GST calculations, payout splits) where silent type errors are dangerous.
- Faster initial development velocity for a solo engineer.

Go was seriously considered. It would give better raw performance and lower EC2 memory usage. Revisit Go for high-throughput background workers (search sync, payout batch jobs) if Node becomes a bottleneck — but do not migrate the main API unless there is a measured performance problem.

Python was rejected: slower runtime, worse TypeScript-equivalent type safety, and weaker Fastify/Express equivalent ecosystem.

### Framework: Fastify

**Chosen over:** Express, NestJS, Hono

Fastify gives:

- 2-3x higher throughput than Express in benchmarks
- Built-in JSON Schema validation on request/response — critical for enforcing correct input on financial endpoints
- TypeScript support without additional configuration
- Plugin system that encourages clean separation of concerns

NestJS was rejected: too much boilerplate and "magic" for a solo engineer. Debugging decorator-based DI issues costs time we do not have.

Hono was considered but has a smaller ecosystem and less battle-tested production use at the time of this decision.

### ORM strategy: Prisma for migrations, raw SQL for queries

Prisma is used **only** for:

- Schema definition (the source of truth for table structure)
- Migration file generation and execution

All database queries use the raw `pg` driver directly.

Reason: Prisma's query builder generates inefficient SQL for joins across 3+ tables. Construction order queries routinely join orders → sub_orders → order_items → product_variants → supplier_products → pricing_tiers. Letting Prisma generate these would produce N+1 queries. Raw SQL gives us full control and makes query performance explicit and reviewable.

### Database: PostgreSQL 15 on AWS RDS

**Chosen over:** MySQL, MongoDB, PlanetScale

PostgreSQL was the only serious option for this domain:

- Construction product data is highly relational. Categories have depth. Pricing has tiers. GST has rates per HSN code with effective dates. A document store makes these joins painful.
- ACID compliance is non-negotiable for order and payment records.
- `pgvector` extension available for future ML-based recommendation embeddings without adding a separate vector database.
- Row-level security available for multi-tenant supplier data isolation if needed.

AWS RDS gives us managed backups, automatic failover (Multi-AZ), and read replicas without operational overhead.

### Cache and queue: Redis on AWS ElastiCache

Used for:

- Session token storage
- API rate limiting counters
- Product listing cache (TTL-based)
- BullMQ job queues for background workers

A single `cache.t3.micro` node at launch. Cluster mode when recommendation serving requires sub-millisecond latency.

### Search: OpenSearch on AWS (managed)

**Chosen over:** Algolia, Typesense, PostgreSQL full-text search

OpenSearch gives:

- Fuzzy matching with configurable edit distance (handles "sement" → "cement", "saraia" → "sariya")
- Phonetic analysis plugin for Hindi product names in Roman script
- `function_score` query for combining text relevance with sale/stock/popularity ranking signals
- Synonym dictionary for Indian construction vocabulary (sariya, gitti, bajri, OPC, PPC, CP fittings)
- Managed on AWS — no server maintenance

Algolia rejected: simpler API but costs become prohibitive at scale and lacks phonetic analysis for Hindi terms.

PostgreSQL full-text search rejected: no fuzzy matching, no synonym support, no custom ranking formula. Would hit its ceiling within 2 months.

## Consequences

- New engineers need Node.js / TypeScript experience
- Complex database queries are raw SQL — engineers must be comfortable reading and writing SQL joins
- OpenSearch adds operational complexity — the outbox sync pattern must be maintained correctly (see ADR-004)
- If the team grows and Go engineers join, background workers can be migrated to Go independently of the main API
